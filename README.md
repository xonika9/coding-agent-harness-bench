# Coding agent harness bench

Я собирал этот стенд не ради ещё одного рейтинга моделей. Мне нужно было выбрать
кодинг-агента для себя и понять, сколько на самом деле решает обвязка вокруг модели.

Внутри четыре небольшие изолированные задачи на Python и TypeScript. Это не
production-проекты: каждая решается за несколько минут. Сложность здесь в строгих
условиях и hidden tests, которые ломают поверхностные решения. Я прогнал на них семь
связок `harness × model × effort` 16 июля 2026 года.

## Коротко

Всего получилось 56 headless-прогонов: по два прогона каждой связки на каждой задаче.
Все 56 прошли видимые и скрытые тесты. Поэтому сравнивать пришлось не правильность, а
цену и время.

Главное сравнение — одна модель и один effort в трёх разных harness:

| harness | model | effort | оценка за 8 прогонов | wall time |
|---|---|---|---:|---:|
| Codex | `gpt-5.6-sol` | `medium` | $3,77 | 19,1 мин |
| opencode | `gpt-5.6-sol` | `medium` | $5,28 | 26,8 мин |
| omp | `gpt-5.6-sol` | `medium` | $23,06 | 84,5 мин |

В моих прогонах omp на той же модели вышел в 6,11 раза дороже Codex. В расчёте есть
известный недосчёт cache-write у Codex и opencode. Если заложить максимально невыгодную
поправку, разрыв сжимается до 5,38 раза. Точнее уже не посчитать: отдельные
cache-write tokens я не сохранил.

Сам отчёт можно открыть как [HTML](report.html) или [PDF прямо в GitHub](report.pdf).
Цифры лежат в одном файле:
[report-data/metrics-710.json](report-data/metrics-710.json).

## Что лежит в репозитории

```text
tasks/t7..t10/
  PROMPT.md       задание агенту
  fixture/        стартовый рабочий каталог
  heldout/        скрытые тесты для локального grader
  grade.sh        проверка результата
report-data/
  metrics-710.json
report.html
report.pdf
ENVIRONMENT.md
```

Готовых решений здесь нет. Они только подсказывали бы агентам ответы и раздували
репозиторий. Каталоги всех 56 прогонов я тоже не оставлял. Из-за этого итоговые оценки
можно проверить по датасету, но нельзя независимо пересчитать из первичных usage-событий.

## Задачи

| ID | Стек | Что строит агент | Видимых + скрытых тестов |
|---|---|---|---:|
| `t7-ts-formula` | TypeScript | движок формул: токенайзер, парсер, функции, ошибки | 61 + 45 |
| `t8-py-query` | Python | filter → sort → distinct → paginate → project | 29 + 22 |
| `t9-ts-vm` | TypeScript | интерпретатор стековой VM | 30 + 33 |
| `t10-py-cron` | Python | вычислитель cron-расписаний | 19 + 22 |

`heldout/` открыт, чтобы grader можно было проверить и запустить самому. Агент не должен
видеть эти тесты. Поэтому не запускайте его из корня репозитория: сначала скопируйте
`fixture/` во временный каталог.

## Как повторить один прогон

```bash
TASK=tasks/t7-ts-formula
WD="$(mktemp -d)"
cp -R "$TASK/fixture/." "$WD/"

git -C "$WD" init -q
git -C "$WD" add .
git -C "$WD" \
  -c user.name=bench \
  -c user.email=bench@example.invalid \
  commit -qm baseline

PROMPT="$(cat "$TASK/PROMPT.md")"
```

Теперь запустите в `$WD` любой из harness:

```bash
# Codex
CLEAN_CODEX_HOME="${TMPDIR:-/tmp}/codex-bench-home"
mkdir -p "$CLEAN_CODEX_HOME"
ln -sf "$HOME/.codex/auth.json" "$CLEAN_CODEX_HOME/auth.json"

# В новом CODEX_HOME нет глобальных MCP. Проверка должна вывести пустой список.
CODEX_HOME="$CLEAN_CODEX_HOME" codex mcp list

CODEX_HOME="$CLEAN_CODEX_HOME" codex exec \
  --sandbox workspace-write \
  -c approval_policy="never" \
  -m gpt-5.6-sol \
  -c model_reasoning_effort=medium \
  -C "$WD" "$PROMPT" < /dev/null

# opencode
opencode run --pure --auto \
  -m openai/gpt-5.6-sol \
  --variant medium \
  --format json \
  --dir "$WD" "$PROMPT"

# omp
omp -p \
  --model openai-codex/gpt-5.6-sol \
  --thinking medium \
  --mode json \
  --no-lsp --no-session \
  --cwd "$WD" "$PROMPT"
```

После прогона вернитесь в корень репозитория и запустите grader:

```bash
bash "$TASK/grade.sh" "$WD"
```

Для TypeScript понадобятся `node` и `tsc`, для Python — только `python3`. Версии, на
которых я делал исходные прогоны, записаны в [ENVIRONMENT.md](ENVIRONMENT.md).

## Как считалась стоимость

Все инструменты у меня были на подписках. Доллары в таблице не списывались с карты: это
оценка стоимости тех же токенов по API-тарифам.

Для `codex*` и `oc*` использовалась формула:

```text
cost = ((input - cached_input) * 5
        + cached_input * 0.5
        + output * 30) / 1e6
```

Здесь весь некэшированный вход считается по $5/M. Но [cache writes у OpenAI стоят
1,25× от обычного input](https://developers.openai.com/api/docs/models/gpt-5.6-sol), то
есть $6,25/M. Отдельного счётчика cache-write в логах не осталось.

Эта погрешность есть только у `codex*` и `oc*`. Для omp я брал `usage.cost.total` с
дедупликацией по `responseId`, для Claude Code — `total_cost_usd`. Если представить
самый плохой для Codex случай и считать, что весь свежий вход сразу записывался в кэш,
получится так:

| Связка | Датасет | Консервативный максимум |
|---|---:|---:|
| `codex56` | $3,77 | $4,29 |
| `oc56` | $5,28 | $6,27 |
| `omp56 / codex56` | 6,11× | 5,38× |

Реальный результат где-то между этими границами: часть входа была свежей, часть читалась
из кэша по $0,50/M.

Ещё одна поправка: `ccusage-codex` вернул для `t9/r1/codex56` fallback $0,125. Эта цифра
не сходится даже на уровне арифметики. Только 5320 выходных токенов стоят $0,1596, ещё
до учёта входа. Пересчёт дал $0,4479 — его я и записал в `metrics-710.json`.

## Что я выбрал

Я остаюсь на Codex. Связка Codex + GPT выглядит именно так, как должна выглядеть
вендорская пара модели и harness: быстро, экономно, без лишней возни.

Если когда-нибудь придётся отказаться от GPT или Claude, запасной вариант у меня уже
есть: opencode с подпиской Go за $10 и доступом к основным open source моделям.

omp 16.5.2 с GPT-5.6 medium для меня оказался переоценённой хуетой. Он проиграл на каждой
задаче и по цене, и по времени. Это вывод о конкретной протестированной конфигурации, но
для своей работы я её больше не рассматриваю.

Claude Code с Opus тоже не убедил: дорого и долго даже с xhigh effort. Codex с GPT-5.5
high на этих же задачах отрабатывал заметно быстрее.

## Где тесту не стоит верить слишком сильно

- Четыре задачи и два прогона — мало для разговора о надёжности. Здесь имеет смысл
  смотреть на разницу в разы, а не на десятые доли.
- Effort у всех семи колонок разный. Чисто сравнивать между собой можно прежде всего
  `codex56`, `oc56` и `omp56`: там совпадают и модель, и effort.
- Я запускал задачи параллельно (`-P4`/`-P6`), поэтому абсолютное время немного шумит.
- Для восьми ячеек Codex на t7/t8 время восстановлено по таймстемпам. В датасете они
  помечены как `wall_src: proxy`; это нижняя оценка.
- Первичных input/cache/cache-write tokens и каталогов прогонов в публичном репозитории
  нет.
- Это снимок конкретных версий на 2026-07-16. Через несколько обновлений цифры наверняка
  будут другими.
