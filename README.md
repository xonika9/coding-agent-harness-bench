# Личный тест-стенд харнессов

Не бенчмарк. Личная история: «что лучше именно мне на моём стеке». Стенд гоняет
одинаковые задачи через четыре связки, снимает объективные цифры и слепое
качественное ревью, а решение принимается по заранее зафиксированному правилу —
чтобы не подгонять вывод под ощущение.

## Четыре колонки (харнесс × модель × effort)

| ключ | харнесс | модель | effort | по чему бьёт |
|---|---|---|---|---|
| `codex56` | Codex | gpt-5.6-sol | medium | инкумбент |
| `ccx` | Claude Code (прокси) | gpt-5.6-sol | medium | тот же движок, что у Codex |
| `ccopus` | Claude Code (нативный) | opus-4.8 | xhigh | чистый Claude, без прокси |
| `codex55` | Codex | gpt-5.5 | high | как работал раньше |

Три попарных среза из одних и тех же чисел: `codex56↔ccx` = эффект харнесса (тот твит);
`ccx↔ccopus` = эффект модели внутри CC; `codex56↔ccopus` = два цельных продукта в лоб.
**Эффорты разные по личному выбору** — это «мои реальные конфиги», а не «модели при
равном усилии». Так и подаём в статье.

## Окружение (зафиксировано)

- **Claude Code** v2.1.210 — колонки `ccx`, `ccopus`
- **OpenAI Codex** v0.144.4 — колонки `codex56`, `codex55`
- Тулчейн грейдеров: Node 26.5.0 (нативный `.ts` + `node --test`), Python 3.14.6
  (`unittest`), TypeScript 5.9.3 (`tsc`)
- Дата прогона: _вписать при запуске_

## Шесть задач

| | стек | режим | оракул | сложность |
|---|---|---|---|---|
| `t1-py-bugfix` | Python | fix (игрушечный) | unittest зелёный + тесты не тронуты | лёгкая |
| `t2-ts-refactor` | TS | refactor по 12 файлам | tsc чист + node:test + нет старой сигнатуры | средняя |
| `t3-ts-feature` | TS | build по спеке | видимые + held-out тесты | средняя |
| `t4-diagnosis` | Python | diagnose | `ANSWER.txt` = `file:line` (±1) | лёгкая |
| `t5-diff-bugfix` | Python | fix на **реальном коде** (модуль из моего рабочего проекта, `_diff.py`) | настоящие `test_diff.py` (11) зелёные | средняя |
| `t6-ts-feature` | TS | build многошаговый (мини-DSL фильтра) | видимые + held-out тесты | средняя |

Защита от накрутки: правки тестов запрещены и ловятся `git diff baseline -- test/`;
в фиче-задачах есть **скрытые** тесты, применяемые только на грейдинге; `t5` гоняет
твои настоящие ассерты (pytest в системе нет — их запускает `_run_pytests.py`).

## Как прогнать

Всё zero-install: Python `unittest`, `node --test` по `.ts` нативно, `tsc` для типов,
таймаут на `perl`. Правки агентов идут в одноразовые git-клоны под `runs/` (в `.gitignore`).

**0. Один раз — проверить автономный запуск всех четырёх:**
```bash
bash scripts/preflight.sh          # правь scripts/harness.env, пока у всех не будет PING.txt OK
```
`harness.env` — единственное, что нужно выверить руками: там команды запуска
(модель-слаги, effort-флаги Codex, что прокси поднят на :8318). preflight это докажет.

**Прогон одной задачи (пример, RUN=1):**
```bash
RUN=1 bash scripts/setup-task.sh   t6-ts-feature      # 4 свежих клона
# запусти 4 колонки — в 4 терминалах параллельно (тогда видно, кто быстрее) или подряд:
RUN=1 bash scripts/run-one.sh t6-ts-feature codex56
RUN=1 bash scripts/run-one.sh t6-ts-feature ccx
RUN=1 bash scripts/run-one.sh t6-ts-feature ccopus
RUN=1 bash scripts/run-one.sh t6-ts-feature codex55
RUN=1 bash scripts/grade-all.sh     t6-ts-feature     # -> results/…/r1/grades.tsv
RUN=1 bash scripts/harvest.sh       t6-ts-feature     # -> results/…/r1/metrics.tsv
RUN=1 bash scripts/collect-blind.sh t6-ts-feature     # -> слепые диффы A–D + скрытый mapping
```
Второй прогон — то же с `RUN=2` (клоны и результаты не пересекаются). Затем следующая задача.

Ничего сторожить не надо: `--dangerously-skip-permissions` / `--dangerously-bypass-…`
делают запуск автономным, таймаут (`TIMEOUT_SEC`, по умолчанию 1200с) режет зависшие.

## Цифры, которые снимаем

- **success (0/1)** — скриптом-оракулом. Первично, решает всё остальное — тай-брейк.
- **wall_s** — из обёртки раннера (кто быстрее — только среди прошедших оракул).
- **tokens in/out, turns** — из `claude -p --output-format json` (ccx, ccopus) и из
  `codex exec --json` (codex56, codex55).
- **$ (API-equivalent)** — все связки на подписках, реального пофичного счёта нет; это
  «сколько стоило бы на API». Для Opus даёт сам claude-json; для Codex — сверка
  `ccusage-codex session --json`; CodexBar — живой индикатор квоты.
- **cap_hit** — вручную: упёрся ли `ccx` в 200K и коряво ли закомпактил (структурная
  засада, выносится отдельно от «модель хуже»).

## Правило решения (зафиксировано заранее)

task-score связки = сумма по задачам от (успешных прогонов / 2), диапазон 0–6.

1. **Success-гейт.** Challenger бьёт инкумбента только при разрыве task-score **≥ 1.0**.
   ≤ 0.5 — ничья, в тай-брейк.
2. **Клапан против шума.** Ячейка расщепилась 1–1 и стоит в 0.5 от переворота — добавить
   3-й прогон только для неё.
3. **Тай-брейк:** меньше вмешательств → быстрее → меньше ходов → меньше ретраев правок.
4. **Инкумбент.** Codex остаётся, если challenger не взял гейт или не выиграл тай-брейк
   уверенно + не нужна экосистема CC.
5. **cap_hit** у `ccx` как единственная причина провалов — отдельной строкой.

Слепое качественное ревью — по `results/<task>/r<RUN>/blind/{A..D}.diff`; расшифровка
`mapping.tsv` открывается только после того, как разбор написан.

## Честные ограничения

Малая выборка (6×4×2) — доверять только **крупным** разрывам; тонкую разницу «на 8%» тест
не ловит, и она всё равно не повод уходить с рабочего инструмента. Эфорты не выровнены
(так задумано). Межвендорные токены как «цена» не сравниваются — для этого $ API-equiv.
`t5` вшивает снимок реального `_diff.py`/`test_diff.py` — перед публикацией в блог
подтверди, что эти сниппеты не жалко показывать (по разведке — generic-утилиты, безопасно).
